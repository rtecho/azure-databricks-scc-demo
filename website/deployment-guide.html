Deployment Guide

Step-by-step instructions to deploy Azure Databricks with Secure Cluster Connectivity

Prerequisites

Before deploying, ensure the following requirements are met.

  • Azure subscription with Contributor role
  • Azure CLI 2.50+ with Bicep extension
  • Resource provider registered: Microsoft.Databricks
  • Premium tier selected (required for Private Link features)
  • VNet planning completed (CIDR ranges decided)

Option 1: Deploy via Azure Portal

Follow these steps to deploy through the Azure portal UI.

  1. Step 1 — Create Resource Group

    Navigate to the Azure portal and create a new resource group in your target region. Use a clear naming convention such as rg-databricks-scc-demo.

  2. Step 2 — Create Azure Databricks Workspace
    • Search "Azure Databricks" in the portal marketplace
    • Select the Premium pricing tier
    • On the Networking tab: set "Deploy with Secure Cluster Connectivity (No Public IP)" to Yes (this is the default)
    • Select "Use your own VNet" for VNet injection
    • Configure the host subnet and container subnet (minimum /26 each, delegated to Microsoft.Databricks/workspaces)
  3. Step 3 — Verify Deployment
    • Check the workspace is in Active state in the Azure portal
    • Launch the workspace and create a test cluster
    • Verify the VM has no public IP in the managed resource group

Option 2: Deploy via Azure CLI

Use the Azure CLI to deploy programmatically. This approach is ideal for repeatable deployments and CI/CD pipelines.

# Register provider
az provider register --namespace Microsoft.Databricks

# Create resource group
az group create --name rg-databricks-scc-demo --location southeastasia

# Deploy using Bicep template
az deployment group create \
  --resource-group rg-databricks-scc-demo \
  --template-file infra/main.bicep \
  --parameters infra/main.parameters.json

Option 3: Deploy via Bicep Template

For infrastructure-as-code deployments, use the Bicep template directly. The key resource configuration for the Databricks workspace with SCC enabled is shown below.

resource workspace 'Microsoft.Databricks/workspaces@2024-05-01' = {
  name: workspaceName
  location: location
  sku: { name: 'premium' }
  properties: {
    managedResourceGroupId: managedRgId
    publicNetworkAccess: 'Enabled'
    requiredNsgRules: 'AllRules'
    parameters: {
      enableNoPublicIp: { value: true }
      customVirtualNetworkId: { value: vnet.id }
      customPublicSubnetName: { value: 'snet-host' }
      customPrivateSubnetName: { value: 'snet-container' }
    }
  }
}

The critical parameter is enableNoPublicIp: { value: true }. This single setting activates Secure Cluster Connectivity, ensuring all cluster nodes are provisioned without public IP addresses.

Verification Checklist

After deployment completes, walk through each verification step to confirm SCC is properly configured.

  • Workspace shows "Active" state in Azure portal
  • Workspace URL is accessible: https://adb-{workspace-id}.azuredatabricks.net
  • Create a test cluster (2 nodes)
  • In managed resource group, check VM has no Public IP
  • In VNet, verify NSG rules are auto-provisioned
  • Verify subnet delegation shows Microsoft.Databricks/workspaces
  • NAT Gateway shows associated with both subnets
  • Test cluster runs a simple notebook: spark.range(100).count()

Upgrade Existing Workspace

You can enable Secure Cluster Connectivity on an existing Azure Databricks workspace without recreating it. Follow the steps below carefully.

  1. Stop all compute resources first

    Terminate all running clusters, jobs, and SQL warehouses before making network changes.

  2. Via Azure Portal

    Navigate to your workspace resource → Settings → Networking → Enable Secure Cluster Connectivity (No Public IP).

  3. Via ARM Template

    Set enableNoPublicIp to true in the workspace properties and redeploy.

  4. Via Terraform

    Set no_public_ip = true in the custom_parameters block (requires azurerm provider version 3.41.0+).

The network update takes approximately 15 minutes to propagate. During this time, new clusters cannot be started. Temporary rollback is possible by setting enableNoPublicIp back to false.

Troubleshooting

Common issues encountered when deploying or enabling SCC, and how to resolve them.

Issue Cause & Resolution
Cluster fails to start after enabling SCC Check that NSG rules have not been manually modified. Databricks auto-provisions required NSG rules on both subnets — manual edits can break relay connectivity. Also verify that service endpoints for Microsoft.Storage and Microsoft.Sql are configured on both subnets.
Cannot reach external services Ensure a NAT Gateway is properly associated with both the host subnet and the container subnet. Without a NAT Gateway (or other explicit outbound method), cluster nodes have no path to the internet for package downloads, PyPI, or external APIs.
Subnet delegation error Both subnets must be exclusively delegated to Microsoft.Databricks/workspaces. They cannot be shared with other Azure services (e.g., App Service, Container Instances). Create dedicated subnets for Databricks.
Private endpoint DNS resolution fails Configure private DNS zones for privatelink.azuredatabricks.net and link them to your VNet. If using custom DNS servers, add conditional forwarders for the azuredatabricks.net zone pointing to Azure DNS (168.63.129.16).

Cleanup

When you are finished with the demo, delete the resource group to remove all deployed resources and stop incurring charges.

az group delete --name rg-databricks-scc-demo --yes --no-wait

This command deletes the resource group and all resources within it, including the Databricks workspace, the managed resource group, VNet, NAT Gateway, and any associated resources. The --no-wait flag returns immediately while deletion proceeds in the background.
