Network Design & Configuration

VNet injection, NSG rules, private endpoints, and egress control

VNet Injection Overview

VNet injection deploys Azure Databricks compute resources directly into the customer's own Azure Virtual Network. Rather than running clusters in a Databricks-managed VNet, your Spark driver and executor VMs are placed in subnets you control, giving you full visibility and governance over network traffic.

This model enables you to apply your own network security policies, use service endpoints and private endpoints, route traffic through firewalls or NVAs, and peer with other VNets in your organization.

Requirements

  • The VNet and Databricks workspace must be in the same Azure region and subscription.
  • VNet address space must be between /16 and /24 CIDR.
  • Two dedicated subnets are required — they cannot be shared with other workspaces or any other Azure resources.
  • Both subnets must be delegated to Microsoft.Databricks/workspaces.
  • Minimum subnet size: /26 (Databricks does not recommend smaller than /26).

Subnet Planning

Plan your subnet layout carefully before deployment. Each Databricks workspace requires its own pair of dedicated subnets. The following table shows the standard naming conventions and purposes.

Subnet Name Convention CIDR Purpose Delegation
Host (Public) snet-dbw-host /24 or /26 Hosts cluster node VMs Microsoft.Databricks/workspaces
Container (Private) snet-dbw-container /24 or /26 Container networking for Spark executors Microsoft.Databricks/workspaces
Private Endpoint snet-dbw-pe /27 Private endpoints (front-end, back-end) None

Plan Carefully — Subnets Cannot Be Resized

Subnet CIDR ranges cannot be changed after deployment. Plan carefully! Azure reserves 5 IPs per subnet. Each cluster node needs 1 IP in the host subnet + 1 IP in the container subnet.

NSG Rules for SCC

Azure Databricks automatically provisions and manages Network Security Group (NSG) rules via subnet delegation. These rules are required for the service to function. Do NOT modify or delete the auto-provisioned rules.

With Secure Cluster Connectivity enabled, there are no inbound rules for ports 22 or 5557. Those inbound rules only exist when SCC is disabled (legacy mode). The control plane communicates through the outbound relay tunnel instead.

Direction Protocol Source Destination Ports Purpose
Inbound Any VirtualNetwork VirtualNetwork Any Default internal
Outbound TCP VirtualNetwork AzureDatabricks 443, 3306, 8443-8451 Control plane + relay
Outbound TCP VirtualNetwork Sql 3306 Metastore, system tables
Outbound TCP VirtualNetwork Storage 443 DBFS, artifact store
Outbound Any VirtualNetwork VirtualNetwork Any Internal cluster traffic
Outbound TCP VirtualNetwork EventHub 9093 Structured streaming, logs

SCC Eliminates Inbound Control Plane Ports

Notice: With SCC enabled, there are NO inbound rules from the AzureDatabricks service tag on ports 22 or 5557. The control plane uses the outbound relay tunnel instead.

Do NOT Modify Auto-Provisioned NSG Rules

Do NOT delete or modify the auto-provisioned NSG rules. They are managed by Azure Databricks via subnet delegation. Use a separate NSG if you need custom rules.

NAT Gateway Configuration

A NAT Gateway provides a stable, predictable egress IP address for outbound traffic from your Databricks clusters. This is essential for firewall allowlisting and downstream service access control. After March 31, 2026, new Azure VNets will require an explicit outbound connectivity method — default outbound access will be retired. NAT Gateway is the recommended solution for VNet-injected Databricks workspaces with SCC enabled.

Configuration Requirements

  • Associate the NAT Gateway with both the host subnet and the container subnet.
  • Use a Standard SKU public IP for predictable, stable egress.
  • A single NAT Gateway can serve both subnets in the same workspace.

Benefits

Stable Egress IP

Provides a fixed public IP for all outbound traffic, making it easy to allowlist in downstream firewalls and third-party services.

SNAT Port Management

NAT Gateway handles SNAT port allocation automatically, avoiding port exhaustion issues that can occur with Azure Load Balancer default SNAT.

Required for SCC + VNet Injection

With SCC and VNet injection, cluster nodes have no public IPs. NAT Gateway is the supported method for outbound internet connectivity. Required for new VNets after March 2026.

Private Link Options

Azure Databricks supports three types of Private Link connectivity, each securing a different communication path. These can be combined for maximum network isolation.

Inbound (Front-end) Private Link

Secures user and API access to the workspace. Requires a databricks_ui_api private endpoint plus a browser_authentication endpoint for SSO. Set publicNetworkAccess: Disabled and requiredNsgRules: NoAzureDatabricksRules.

Classic (Back-end) Private Link

Secures control-plane-to-cluster connectivity via a private endpoint instead of the SCC relay. The private endpoint replaces the public relay path, keeping all management traffic within the Azure backbone via private networking.

Outbound (Serverless) Private Link

Secures serverless compute access to customer Azure resources. Uses Network Connectivity Configurations (NCCs) to establish private connectivity from Databricks-managed serverless infrastructure to your data sources.

Combine SCC with Private Link for Complete Isolation

You can combine SCC with Private Link for complete isolation. Front-end Private Link + Back-end Private Link + SCC gives maximum network security. This eliminates all public network paths for both user access and control plane communication.

Service Endpoints

Service endpoints on Databricks subnets enable optimized, direct connectivity to Azure platform services over the Azure backbone network. The following service endpoints should be configured on both the host and container subnets.

Service Endpoint Purpose Required
Microsoft.Storage DBFS root storage and artifact store access Yes
Microsoft.Sql Hive metastore and system tables Yes
Microsoft.EventHub Log delivery and structured streaming Yes
Microsoft.KeyVault Secret management and key-backed encryption Optional

Best Practices

Follow this checklist to ensure a secure, well-architected Databricks network deployment.

  • Always use Premium tier for full SCC + Private Link support
  • Size subnets for growth — /24 recommended over /26
  • Deploy NAT Gateway on both host and container subnets
  • Use separate NSGs (one per workspace recommended)
  • Enable SCC (enableNoPublicIp = true) on all workspaces
  • Plan for Private Link if compliance requires no public access
  • Test with a small cluster before production deployment
  • Document your CIDR allocations — subnets cannot be resized after deployment
