When to Use Each

Select the deployment model that best matches your organization's security requirements, compliance needs, and operational capacity.

Default (Managed VNet)

Best for: Dev/test, non-sensitive data, quick demos

Pros: Zero setup, fully managed

Cons: No network control, public IPs exposed, no compliance features

VNet Injection

Best for: Organizations needing on-premises connectivity, custom DNS, traffic inspection

Pros: Full VNet control, UDR support, service endpoints

Cons: Public IPs still assigned, inbound ports 22/5557 open

VNet Injection + SCC

Best for: Production workloads, regulated industries, security-conscious organizations

Pros: No public IPs, no inbound ports, relay-based secure tunnel

Cons: NAT Gateway required (small cost), workspace access still public

VNet Injection + SCC + Private Link

Best for: Financial services, healthcare, government, maximum isolation

Pros: Complete private access, no public exposure at all, meets strictest compliance

Cons: Most complex setup, Private Link per-endpoint cost, DNS complexity

Migration Path

Organizations typically adopt a progressive approach, starting with the simplest model and upgrading as security requirements evolve. The path below shows the recommended upgrade sequence.

Default → VNet Injection → SCC → Private Link

Default → VNet Injection

Deploy a new workspace with a custom VNet. This transition requires creating a new workspace or re-creating the existing one, as VNet injection cannot be enabled on an already-deployed default workspace. Plan for data and notebook migration.

VNet Injection → SCC

Set enableNoPublicIp = true in the workspace properties. This is an in-place upgrade that takes approximately 15 minutes of downtime. Ensure a NAT Gateway is attached to both subnets before enabling. All running clusters must be terminated first.

SCC → Private Link

Add private endpoints for the workspace (front-end and back-end), configure private DNS zones (privatelink.azuredatabricks.net), and set publicNetworkAccess = Disabled. This is the most involved transition and requires careful DNS planning, especially in hub-spoke network topologies.
