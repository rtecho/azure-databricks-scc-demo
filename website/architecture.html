Control Plane vs Compute Plane

Azure Databricks separates infrastructure into two distinct planes. Understanding each plane's role is essential to understanding how SCC secures the architecture.

Aspect Control Plane Compute Plane
Managed by Databricks (Microsoft-managed) Customer (in customer subscription)
Contains Web app, REST API, cluster manager, job scheduler, notebooks Spark driver & executors, DBFS mount, cluster VMs
Network Databricks-managed VNet Customer VNet (with VNet injection)
Public IPs Yes (service endpoints) No (with SCC enabled)
Inbound ports 443 (HTTPS for users) None (with SCC)
Communication Sends commands via SCC relay Initiates outbound connection to relay

Data Flow with SCC

The following steps describe the end-to-end data and control flow when Secure Cluster Connectivity is enabled.

  1. User Access — Users connect to the Databricks workspace URL via HTTPS (443). This goes to the control plane.
  2. Cluster Creation — Control plane instructs Azure to provision VMs in the customer's VNet. VMs get private IPs only (no public IPs).
  3. Relay Connection — Each cluster node initiates an outbound HTTPS connection to the SCC relay endpoint on port 443. Uses a different IP than the web app.
  4. Command Tunnel — The control plane sends cluster administration commands (start, stop, configure) through the relay tunnel. No direct SSH needed.
  5. Data Access — Cluster nodes access data sources (DBFS, ADLS, SQL) via service endpoints or private endpoints within the VNet.
  6. Egress — For internet-bound traffic, the NAT Gateway provides a stable egress IP. All Azure-bound traffic stays on the backbone.

Managed Resource Group

When you create an Azure Databricks workspace, the service automatically creates a managed resource group with the naming convention databricks-rg-{workspace}-{unique}. This resource group is fully managed by the Databricks platform and exists within your Azure subscription.

What It Contains

The managed resource group holds the infrastructure for your clusters: virtual machines (VMs), managed disks, network interfaces (NICs), and network security groups (NSGs) for cluster nodes.

Customer Restrictions

Customers cannot modify, delete, or add resources in the managed resource group. Azure Policy deny assignments prevent changes. Databricks manages the full lifecycle of these resources.

VNet Injection Behavior

With VNet injection, cluster VMs are placed in the customer's own VNet subnets (host and container subnets) but are still tracked and managed within the managed resource group.

Key Architecture Decisions

SCC is enabled via enableNoPublicIp = true in the workspace properties. This is the single toggle that activates the entire secure connectivity model.

Two subnets are required for VNet injection: a host subnet (public) and a container subnet (private). Both must be delegated to Microsoft.Databricks/workspaces. Minimum size /26 each.

NAT Gateway is strongly recommended for VNet injection with SCC to provide stable egress IPs. After March 31, 2026, new Azure VNets will require explicit outbound connectivity methods.
